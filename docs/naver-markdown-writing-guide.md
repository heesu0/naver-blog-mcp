# 네이버 블로그용 Markdown 작성 가이드

이 문서는 Markdown 원문을 네이버 블로그에 올릴 때, 현재 `heesu0/naver-blog-mcp` 변환/발행 흐름에서 가장 안정적으로 보이는 작성 규칙을 정리한 가이드입니다.

## 기본 원칙

- 원문은 Markdown으로 작성한다.
- 제목은 frontmatter `title`과 본문 H1을 맞춰 쓴다.
- 태그는 기본적으로 쓰지 않는다.
- 클릭 목차는 만들지 않는다. 네이버가 내부 anchor/id를 제거한다.
- 목차는 읽기용 2단계 목록으로 작성한다.
- 이미지, 표, 코드블록은 발행 후 실제 화면에서 확인한다.

## 권장 파일 구조

```md
---
title: 글 제목
---

# 글 제목

도입 문단입니다.

## 목차

- 첫 번째 큰 주제
  - 하위 항목
  - 하위 항목
- 두 번째 큰 주제
  - 하위 항목
  - 하위 항목

## 첫 번째 큰 주제

본문입니다.

## 두 번째 큰 주제

본문입니다.
```

## Frontmatter

### 권장

```md
---
title: 글 제목
---
```

### 비권장

```md
---
title: 글 제목
tags:
  - 마크다운
  - 자동화
---
```

현재 MCP는 희수님이 명시적으로 태그를 넘기지 않는 이상 frontmatter `tags`를 네이버 태그로 사용하지 않는다. 하지만 혼동을 줄이기 위해 네이버 블로그용 글에는 `tags`를 작성하지 않는 것을 기본으로 한다.

네이버가 블로그 첫 글 상태에서는 `#첫글` 태그를 자동으로 붙일 수 있다. 이는 MCP나 Markdown 원문에서 넣은 태그가 아니다.

## 제목 규칙

```md
# 글 제목

## 큰 섹션

### 작은 섹션
```

- H1은 글 제목으로 한 번만 사용한다.
- 본문 큰 단락은 H2를 사용한다.
- H3는 필요할 때만 사용한다.
- H4 이하로 깊게 들어가는 구조는 피한다.

## 읽기용 목차

네이버 블로그에서는 HTML anchor 방식의 클릭 목차가 안정적으로 동작하지 않는다.

검증 결과 네이버는 다음 속성을 제거하거나 변환한다.

- `id="section"`
- `name="section"`
- `href="#section"`

따라서 목차는 클릭 이동용이 아니라 읽기용으로 작성한다.

### 권장 목차 형식

```md
## 목차

- 마크다운 작성 흐름
  - 원문 작성
  - frontmatter 관리
  - 이미지/링크 정리
- 네이버용 HTML 변환
  - 제목/문단 변환
  - 목록/표/코드블록 변환
  - 태그 제거
- 발행 전 확인
  - 실제 화면 검증
  - 불필요한 태그 확인
  - 테스트 글 삭제
```

변환 후 네이버에서는 대략 다음처럼 보인다.

```text
• 마크다운 작성 흐름
      ◦ 원문 작성
      ◦ frontmatter 관리
      ◦ 이미지/링크 정리
• 네이버용 HTML 변환
      ◦ 제목/문단 변환
      ◦ 목록/표/코드블록 변환
      ◦ 태그 제거
• 발행 전 확인
      ◦ 실제 화면 검증
      ◦ 불필요한 태그 확인
      ◦ 테스트 글 삭제
```

상위 항목 사이에 빈 줄을 넣지 않아도 된다. 하위 항목은 상위 항목보다 스페이스 2칸 들여쓴다.

```md
- 상위 항목
  - 하위 항목
```

## 목록 작성

### 불릿 목록

```md
- 항목 A
- 항목 B
  - 하위 항목 B-1
```

네이버에서는 `•`, `◦` 형태로 표현된다.

### 번호 목록

```md
1. 첫 번째
2. 두 번째
3. 세 번째
```

번호 목록은 절차나 순서가 중요한 경우에만 사용한다. 목차에는 번호 없는 불릿 목록을 기본으로 사용한다.

## 문단 작성

일반 문단은 그대로 작성한다.

```md
이 문장은 일반 문단입니다. 너무 긴 문단은 모바일에서 읽기 어렵기 때문에 2~4문장 단위로 나누는 것이 좋습니다.
```

권장:

- 한 문단은 너무 길게 쓰지 않는다.
- 모바일 화면 기준으로 중간중간 H2/H3와 목록을 넣는다.
- 문단 사이를 과하게 띄우지 않는다.

## 강조

```md
**굵은 글씨**
*기울임*
`인라인 코드`
```

- 굵은 글씨는 핵심어 강조에 사용한다.
- 기울임은 자주 쓰지 않는다.
- 인라인 코드는 명령어, 파일명, 옵션명, 짧은 코드 표현에 사용한다.

## 코드블록

````md
```ts
const workflow = ['write markdown', 'convert html', 'publish naver'];
console.log(workflow.join(' -> '));
```
````

현재 코드블록은 네이버의 공식 코드블록 컴포넌트가 아니라, HTML 스타일로 코드블록처럼 보이게 변환된다. 화면상으로는 회색 박스 형태로 표시된다.

권장:

- 코드블록은 너무 길게 넣지 않는다.
- 긴 코드는 핵심 부분만 발췌한다.
- 언어 표시는 가능하면 붙인다. 예: `ts`, `js`, `python`, `bash`

## 표

```md
| 단계 | 목적 | 상태 |
|---|---|---|
| Markdown 작성 | 원문 관리 | 완료 |
| HTML 변환 | 네이버 붙여넣기 최적화 | 완료 |
| 블로그 발행 | 실제 화면 검증 | 진행 |
```

표는 네이버에서 테두리 있는 HTML 표로 표시된다.

권장:

- 열은 3~4개 이하로 제한한다.
- 셀 안의 문장은 짧게 쓴다.
- 모바일에서 너무 넓어지는 표는 피한다.

## 인용문

```md
> 좋은 자동화는 글을 대신 망가뜨리지 않습니다. 사람이 읽기 좋은 형태를 유지해야 합니다.
```

네이버에서는 왼쪽 라인과 배경이 있는 박스 형태로 표시된다.

권장:

- 핵심 문장, 주의사항, 요약에 사용한다.
- 여러 문단을 긴 인용문으로 묶는 것은 피한다.

## 링크

```md
[네이버](https://www.naver.com)
```

외부 링크는 일반 링크로 표시된다.

주의:

- 같은 글 내부로 이동하는 `#section` 링크는 사용하지 않는다.
- 네이버가 내부 anchor 링크를 보존하지 않는다.

## 이미지

```md
![이미지 설명](https://example.com/image.png)
```

이미지는 변환 가능하지만, 네이버가 외부 이미지 URL을 어떻게 처리하는지는 상황에 따라 다를 수 있다. 중요한 글에서는 발행 후 반드시 실제 화면을 확인한다.

권장:

- 이미지 alt 텍스트를 작성한다.
- 너무 큰 이미지는 피한다.
- 가능하면 네이버에 직접 업로드되는 방식이 안정적이다. 이 부분은 별도 개선 대상이다.

## 태그

희수님 운영 규칙:

- Markdown에는 `tags`를 쓰지 않는다.
- 네이버 태그도 기본적으로 붙이지 않는다.
- 꼭 필요한 경우에만 발행 요청에서 명시적으로 태그를 전달한다.

현재 MCP 동작:

- frontmatter `tags`는 기본적으로 네이버 태그로 사용하지 않는다.
- 변환 라이브러리가 본문 끝에 붙이는 `#태그` 문단은 제거한다.
- 명시적으로 `tags`를 넘긴 경우에만 네이버 태그 필드에 입력한다.

## 작성 예시

```md
---
title: 마크다운으로 네이버 블로그 글쓰기 자동화하기
---

# 마크다운으로 네이버 블로그 글쓰기 자동화하기

마크다운으로 글을 작성하면 원문 관리가 쉽고, 네이버 블로그 발행 전 검토도 편해집니다.

## 목차

- 작성 흐름
  - 원문 작성
  - 변환 확인
  - 발행 검증
- 변환 결과
  - 제목과 문단
  - 목록과 표
  - 코드블록
- 주의사항
  - 내부 링크 제한
  - 태그 처리
  - 이미지 확인

## 작성 흐름

Markdown 원문을 먼저 작성한 뒤 네이버용 HTML로 변환합니다.

## 변환 결과

목록, 표, 코드블록은 네이버 화면에서 읽기 좋은 형태로 보이도록 변환됩니다.

> 발행 전에는 반드시 실제 네이버 화면에서 확인하는 것이 좋습니다.

## 주의사항

네이버 블로그는 내부 anchor 링크를 보존하지 않으므로 클릭 목차는 사용하지 않습니다.
```

## 요약

네이버 블로그용 Markdown은 복잡한 웹 문서라기보다, 네이버 SmartEditor가 잘 받아들이는 깔끔한 원문으로 작성하는 것이 좋다.

가장 안정적인 형식은 다음과 같다.

- frontmatter에는 `title`만 작성
- H1은 글 제목으로 한 번만 사용
- 목차는 클릭용이 아닌 읽기용 2단계 불릿 목록
- 본문은 H2 중심으로 구성
- 태그는 쓰지 않음
- 코드/표/인용문은 짧고 명확하게 사용
